# Vibe Upskill — 从 Agent 对话中提炼真正的开发能力

## 目标

将用户与 AI Agent 的完整开发对话（需求分析 → 架构设计 → 编码实现 → 排错调试 → 部署上线）转化为**个人可内化的知识**，解决 Vibe Coding 时代"离开 Agent 就不会写代码"的核心痛点。

具体目标：

1. **复盘**：从对话中提炼关键决策、排错思路、架构选型等有价值信息
2. **出题**：生成 10 类面试场景题，帮助用户在面试中能独立回答
3. **沉淀**：将知识结构化存储，形成持续增长的专属题库与避坑清单

## 触发条件

当用户出现以下意图时激活本 Skill：

- 想复盘一次与 Agent 的开发对话
- 想从开发经历中生成面试题
- 想整理避坑清单或知识库
- 提到"vibe coding 复盘"、"面试准备"、"知识沉淀"、"agent 做了什么"等关键词
- 直接粘贴了一段与 Agent 的对话记录并请求分析
- 提供了与 Agent 的某次对话的名称（如"昨天的排错对话"），希望基于那次的对话进行复盘

## 用户画像与冷启动

### 首次使用：自评画像

在第一次启动本 Skill 时，系统会引导用户建立个人画像，用于调整题目难度和复习节奏。

**自评内容**（3 项必填）：

| 维度 | 选项 | 用途 |
|------|------|------|
| 准备阶段 | 基础积累 / 能力提升 / 查漏补缺 | 调整面试题难度分布 |
| 主要技术栈 | 自由填写（如 Python/Django/React） | 生成与用户相关的场景题 |
| 学习目标 | 面试准备 / 技术能力提升 / 项目复盘沉淀 | 确定输出侧重 |

**难度自适应的具体规则**：

| 准备阶段 | 基础 | 进阶 | 高级 | 说明 |
|----------|------|------|------|------|
| 基础积累 | 35% | 50% | 15% | 进阶为主，面试里没人因为基础答得好就让过 |
| 能力提升 | 15% | 55% | 30% | 进阶是主战场，基础用于快速自检，高级用于拔高 |
| 查漏补缺 | 10% | 45% | 45% | 进阶兜底 + 高级看天花板，基础仅做扫盲确认 |

### 冷启动：没有历史对话怎么办？

如果用户尚无 Agent 开发对话记录，不硬凑知识点。Skill 会提供以下最小启动路径：

1. **开启记录习惯**：下次用 Agent 开发时，在对话开头输入 `#记录模式`，提醒自己有意识地保存关键决策点
2. **关注高价值时刻**：重点关注 Agent 的"为什么选择 X 而不是 Y"、"排查了哪些步骤"、"架构抉择的权衡"等信号
3. **设定首次复盘时间**：建议在 3-5 天内完成第一次有意义的 Agent 开发对话后，回到本 Skill 进行复盘
4. **备选：手动构造**：如果你有纯手工完成的开发任务，也可以直接描述给 Skill，手动提供项目背景、关键决策、遇到的问题
5. **体验种子对话**：Skill 内置了一段模拟的 Agent 开发对话（`references/seed-conversation.md`，约 18 轮，覆盖 Docker 排错 + 数据库选型 + Redis 缓存策略 + 架构演进），你可以对它立即执行完整的三模式复盘。发送 **"使用种子对话"** 或 **"用示例对话体验一下"** 即可开始——这是最快理解 Skill 能做什么的方式

> **原则**：宁缺毋滥。一条从真实项目中提炼的知识点，胜过十条臆造的假问题。

## 核心原则

**不是复制 Agent 的行为，而是理解背后的"为什么"。**

- 追问每个决策的**理由**，而不只是记录**做了什么**
- 将具体操作**泛化**为可迁移的方法论
- 用**场景题**的形式组织知识，模拟真实面试环境
- 强制用户**用自己的话复述**，确保真正内化而非"看懂了"

## 完整工作流

### 第零步：确定输入源与复盘范围

首先与用户确认两件事：

**输入源**（支持三种模式）：

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 对话记录模式 | 用户粘贴 Agent 对话历史（聊天导出、截图、复制文本），或直接提供**对话名称**（如"昨天的 Docker 排错对话"）由 Agent 在历史中检索 | 最常用，信息最丰富 |
| 种子对话模式 | 用户说"使用种子对话"或"用示例对话体验"，Skill 加载 `references/seed-conversation.md` 中预置的 18 轮示例对话（Docker 排错 + 数据库选型 + 缓存策略 + 架构演进）作为输入 | 首次用户体验，零门槛立即跑通三模式流程 |
| 实时捕获模式 `[需外部hook支持]` | 在 Agent 会话进行中或刚结束时触发，实时捕获推理过程 | 信息保真度最高，但需要 hook/插件基础设施 |

> **实时捕获模式说明**：该模式依赖外部钩子机制（如 Claude Code 的 hooks 系统、VS Code 扩展的事件监听等）来在 Agent 推理过程中自动记录关键决策点。本 Skill 本身不实现这些钩子——它负责的是"收到捕获数据后如何处理"。如果你使用的 Agent 平台不支持 hooks，推荐的替代方案是**热复盘**：在 Agent 会话结束后**立即**（2 小时内）使用"对话记录模式"触发复盘，此时对话仍在新近记忆中，信息保真度接近实时捕获。

> **对话名称引用**：在"对话记录模式"下，用户无需手动复制粘贴整个对话——只需提供**对话名称**（如"昨天跟 Claude 排 Docker 的那个对话"、"上周三的架构设计讨论"），Agent 会自动在会话历史或对话列表中检索匹配的对话。用户需要在请求中给出足够定位到那次对话的关键词。如果检索到多个匹配项，Agent 会列出候选项请用户确认。

**复盘范围**（必须选择，避免上下文过长导致质量下降）：

按开发阶段分段复盘，每次只处理一个阶段：

- 需求分析阶段
- 架构设计阶段
- 编码实现阶段
- 排错调试阶段
- 部署上线阶段
- 全链路复盘（仅在对话较短时使用）

> **防御机制 — 范围限制**：如果用户提供的对话超过 50 轮交互，必须主动建议分段处理。每段独立生成输出，避免信息过载导致提炼质量下降。

### 模式一：拦截记录 — 从对话中提取关键信息

**目标**：从原始对话中筛选出有价值的"知识信号"，过滤掉无意义的试错和重复操作。

**提取维度**（五个方向，逐一扫描）：

| 维度 | 提取什么 | 信号特征 |
|------|----------|----------|
| 架构决策 | 技术选型、项目结构、设计模式、权衡取舍 | "我们选择 X 因为..."、"对比了 A 和 B" |
| 代码实现 | 关键模式、API 设计、状态管理、数据建模 | Agent 写的核心代码片段、重构过程 |
| 排错过程 | 错误类型、诊断路径、修复策略、预防手段 | 报错信息、Agent 的排查步骤、"原来是..." |
| 部署运维 | 环境配置、CI/CD、容器化、监控 | Docker 命令、部署脚本、配置修改 |
| 方法论 | Agent 如何拆解问题、调研方案、迭代优化 | 任务分解过程、搜索策略、方案对比 |

> **排错维度的可选补充**：如果你手头有对应的 git diff 或 commit（比如 Agent 修 bug 时改的代码），可以一起贴进来帮助还原"具体改了什么"。前提是你觉得有价值——宁缺毋滥，不要为了补充而批量导出 git log。

**对每个知识点，记录以下结构**：

```
【上下文】项目是什么？当前处于什么阶段？
【问题/决策】需要解决什么问题？做什么决策？
【Agent 的做法】Agent 具体做了什么？分几步？
【底层原理】为什么这个做法有效？背后的技术原理是什么？
【替代方案】还能怎么做？为什么选了当前方案？
【可迁移教训】换一个项目，这个经验还适用吗？适用条件是什么？
```

**过滤规则**（丢弃以下内容）：

- 纯重复操作（Agent 试了 5 次同样的方法）→ 只保留最终成功的那次
- 无意义的寒暄和确认（"好的"、"明白了"）
- Agent 的纯机械操作（格式化代码、重命名变量等无决策含量的操作）
- 用户未参与理解的"黑箱"操作 → 标记为 `[待深挖]`

**用户确认环节**：

对于每个提取的知识点，如果 Agent 的决策理由是**推断**出来的（而非对话中明确说明的），必须标注 `[待确认]` 并向用户提问：

> "我注意到 Agent 选择了 Redis 做缓存。对话中没有明确说明原因，我推断是因为需要高频读取且数据结构简单。**[待确认]** 这个推断对吗？你当时是怎么考虑的？"

### 模式二：面试题生成 — 将知识转化为面试场景题

**目标**：基于模式一提取的知识点，生成 10 类面试题，每类至少 1-3 道。

**十类面试题**：

| 编号 | 类型 | 说明 | 示例 |
|------|------|------|------|
| 1 | 场景设计题 | 给定业务场景，要求设计技术方案 | "你的项目需要支持实时协作编辑，如何设计冲突解决机制？" |
| 2 | Bug 排查题 | 给定错误现象，要求分析根因并给出排错路径 | "线上服务返回 502，但容器日志显示正常，你从哪几个方向排查？" |
| 3 | 深挖原理题 | 追问技术选型的底层原因和设计思想 | "为什么选 PostgreSQL 而不是 MongoDB？它的 MVCC 机制是怎么工作的？" |
| 4 | 代码审查题 | 给出一段代码，要求发现问题并提出优化 | "Agent 写的这段并发代码有竞态条件，你能指出来并修复吗？" |
| 5 | 项目表达题 | 要求清晰表达项目的背景、难点和成果 | "请用 3 分钟介绍你的项目，说清楚架构亮点和你解决的最难的问题" |
| 6 | 系统设计题 | 在现有项目基础上做规模扩展设计 | "如果用户量从 100 涨到 10 万，你的数据库架构需要怎么改？" |
| 7 | 运维排障题 | 模拟线上故障场景，考察排查能力 | "凌晨 3 点收到告警，服务 OOM 了，你的排查步骤是什么？" |
| 8 | 方案对比题 | 对比两种技术方案的优劣和适用场景 | "JWT vs Session，你在项目中选了哪个？为什么？什么情况下该选另一个？" |
| 9 | 复盘反思题 | 引导回顾整个项目，提炼改进方向 | "如果重新做这个项目，你会改变哪三个决定？为什么？" |
| 10 | 测试质量题 | 针对核心功能设计测试用例 | "你的用户认证模块，请设计 5 个关键测试用例，覆盖正常流程和边界情况" |

**每道题的输出格式**：

```
### [类型] 题目名称

**难度**：基础 / 进阶 / 高级
**标签**：#架构 #调试 #部署 ...
**来源**：基于 [项目名] 的 [阶段] 阶段提取
**前置知识链**：{列出答好这道题需要先掌握的知识点，标注是否已掌握}

> 示例：要答好这道题，你需要先掌握：
> - Q-015 PostgreSQL MVCC 原理 — 🔴 未复述（建议先复习这道）
> - KB-008 数据库索引 B+Tree 结构 — 🟢 已掌握
> - 坑-003 慢查询 EXPLAIN 分析 — 🟡 复习中（间隔 3 天）

#### 场景描述
（用 2-3 句话描述具体场景，让问题有代入感）

#### 问题
（明确的面试问题）

#### 参考思路
（不是标准答案，而是"应该怎么思考这个问题"的步骤）
1. 首先分析...
2. 然后考虑...
3. 最后...

#### 参考答案要点
（关键知识点，面试时应该提到的核心内容）

#### 反事实推理
（如果条件变化，答案会怎样？）
- 如果把 X 换成 Y，方案需要怎么调整？
- 如果规模扩大 100 倍，瓶颈会出现在哪里？

#### 深挖追问
（面试官可能的追问，帮助用户准备更深层的回答）
- 你提到用 X，那 X 的底层原理是什么？
- 这个方案有什么缺点？你怎么缓解的？
```

**反事实推理 — 强制要求**：

每道题**必须**包含"反事实推理"部分。这是区分"背答案"和"真理解"的关键。通过改变约束条件（规模、技术栈、时间、团队规模等），考察用户是否真正理解了方案的本质。

**用户复述环节 — 强制要求（分层复述）**：

复述分为两层，降低开口的心理门槛：

| 层级 | 要求 | 时长 | 是否必做 |
|------|------|------|----------|
| 骨架复述 | 用 30 秒说清核心思路和关键决策点 | ~30s | **必做** |
| 完整复述 | 展开细节：API 名称、配置参数、具体命令 | ~2min | 选做（建议做） |

每道题生成后，先执行骨架复述：

> "请你用 30 秒，说出这道题的核心思路——不需要细节，只要说清楚'从哪入手、关键决策是什么、为什么'。如果 30 秒说不清，说明你还没真正理解。"

骨架复述通过后，再邀请完整复述：

> "核心思路对了！要不要试试完整复述？补充具体的命令、配置项、API 名称。这一步帮你把'大概懂'变成'能上手做'。当然，跳过也可以，不影响这道题的复习间隔。"

**掌握度自评（四级制，适配分层复述）**：用户复述后需给出自评分数：

| 评分 | 含义 | 触发条件 | 下次复习间隔 |
|------|------|----------|-------------|
| 0 — 完全卡住 | 骨架都没说清，无法回忆核心思路 | 30 秒说不出从哪入手 | 当天内复习 |
| 1 — 骨架通过 | 核心思路正确，但关键细节遗漏或混淆 | 骨架复述通过，完整复述卡住或跳过 | 1 天后复习 |
| 2 — 接近掌握 | 骨架正确，大部分细节能说清，少量遗漏 | 完整复述基本完成，有 1-2 处细节遗忘 | 3 天后复习 |
| 3 — 流畅复述 | 能用自己的话完整讲清思路、原理、细节 | 骨架 + 完整复述均无卡顿 | 7 天后复习 |

> **设计说明**：原三级制（0/1/2）中"1 分"范围太宽——能说核心思路但忘细节，和勉强说出方向但关键点全漏，是两种完全不同的掌握水平。拆成四级后，"骨架通过"成为明确的里程碑：它表示你已经内化了核心判断力，只是还没记熟细节。这在面试中已经具备基本应对能力。评分 3 分起步间隔 7 天（替代原 2 分起步的 3 天），鼓励用户在首次复述时追求完整。

**卡住位置诊断 + 即刻补刀**：当用户标记"卡住了"时，首先标注卡住的**性质**：
- `[概念不清]`：不知道这个概念是什么，需要从定义开始补
- `[逻辑链断裂]`：知道各个概念，但说不清它们之间的因果关系
- `[细节遗忘]`：理解整体思路，但忘了具体 API/配置/参数名

**诊断之后必须立即补刀——不要让用户带着困惑离开**。根据诊断类型，当场生成一个 2 分钟微学习内容：

| 诊断类型 | 即刻补刀内容 | 时长 | 示例 |
|----------|-------------|------|------|
| `[概念不清]` | **微解释**：用类比和最小示例讲清楚这个概念是什么 | ≤ 2min | "MVCC 就像 Git 分支——每个人看到的是自己当时的快照，别人在改你不受影响" |
| `[逻辑链断裂]` | **因果图**：画出 A → B → C 的因果关系链，标注断裂位置 | ≤ 2min | "慢查询 → EXPLAIN 看是 Seq Scan → 没走索引？→ 统计信息过时 → 需要 ANALYZE" |
| `[细节遗忘]` | **速查卡**：列出 3-5 个关键 API/配置项/命令，附带一句话说明 | ≤ 2min | `EXPLAIN ANALYZE` — 显示实际执行计划；`VACUUM` — 回收死元组；`CREATE INDEX CONCURRENTLY` — 不锁表建索引 |

补刀完成后，主动邀请用户立即重新尝试骨架复述：

> "现在试试看——用 30 秒重新说一遍核心思路，看看刚才的缺口补上了没？"

如果补刀后骨架复述通过，评分可以从 0 调整为 1——"你不是不会，你只是缺了一块信息，补上就好了。"

**渐进式提醒机制（适配分层复述）**：

"跳过"的定义调整为——**只做骨架复述不算跳过**。骨架复述是必做的。完整复述可以跳过，不影响复习间隔。

| 连续跳过骨架数 | 行为 |
|---------------|------|
| 第 1-2 次 | 正常标记 `[仅骨架]`，继续下一题 |
| 第 3-5 次 | 主动提醒："你已经连续 3 道题只做了骨架复述。要不要挑一道试试完整复述？只需要 2 分钟补充细节，你的复习间隔能从 1 天直接跳到 7 天。" |
| 第 6-9 次 | 加强提醒："连续 6 道只做骨架——核心思路你都对了，但面试时面试官会追问细节。建议至少把 1 道做完整复述，感受一下差距。" |
| 达到 10 次 | 不硬性暂停（骨架复述本身就是有效学习），但提醒："本会话所有题目均为 `[仅骨架]`。建议你在 24 小时内回来，挑 2 道做完整复述——趁骨架印象还在，补充细节的效率最高。" |

如果用户连骨架复述都拒绝，将该题标记为 `[未复述]`，但不记录为已完成。**跳过骨架 = 跳过整道题**，计数归入原三级提醒（3次提醒 → 6次加强 → 10次硬暂停）。

> **防御机制 — 复述强制**：骨架复述是最低要求。没有经过任何复述（连骨架都没做）的知识点，在最终知识库中标记为"未内化"，在复习提醒中优先出现。超过 10 道完全未复述题目后，必须复述或结束当前复盘会话。仅骨架复述但未完整复述的题目标记为 `[仅骨架]`，复习提醒中次于 `[未复述]` 出现。

### 复习调度引擎（间隔复习系统）

基于简化 SM-2 算法，为每条面试题和知识条目自动计算下次复习时间，确保知识从"短时记忆"转化为"长时记忆"。

**算法规则**：

| 复述评分 | 间隔变化 | 说明 |
|----------|----------|------|
| 0 — 完全卡住 | 重置为 1 天 | 需要重新建立记忆，间隔从头开始 |
| 1 — 骨架通过 | 保持 1 天（不从当前间隔计算） | 核心判断力已内化，细节需要短间隔巩固 |
| 2 — 接近掌握 | 间隔 × 1.5（不低于 1 天） | 大部分掌握，温和延长 |
| 3 — 流畅复述 | 间隔 × 2（上限 120 天） | 记忆稳固，正常延长 |

**间隔序列**：1天 → 3天 → 7天 → 14天 → 30天 → 60天 → 120天（从评分 3 起步时初始间隔为 7 天，替代原评分 2 起步的 3 天）

**首次复述后初始间隔**：
- 骨架通过（评分 1）→ 1 天后复习
- 接近掌握（评分 2）→ 3 天后复习
- 流畅复述（评分 3）→ 7 天后复习

**复习调度表（review-schedule.md）格式**：

```markdown
# 复习调度表

> 最后更新：{日期}
> 当前待复习：{N} 项
> 本周已完成复习：{M} 项

## 🔴 今日到期（{n} 项）
| 编号 | 题目/条目 | 类型 | 上次评分 | 当前间隔 | 来源项目 |
|------|----------|------|----------|----------|----------|
| Q-003 | Docker 容器启动排查 | Bug排查 | 1 | 3天 | Flask应用容器化 |

## 🟡 本周内到期（{n} 项）
| 编号 | 题目/条目 | 类型 | 上次评分 | 到期日期 |
|------|----------|------|----------|----------|
| Q-007 | PostgreSQL vs MongoDB 选型 | 方案对比 | 0 | 2026-06-25 |

## 🟢 已掌握（间隔 ≥60天）（{n} 项）
| 编号 | 题目/条目 | 当前间隔 |
|------|----------|----------|
| Q-001 | Redis 缓存策略 | 120天 |
```

**复习提醒机制**：

- 每次启动 Skill 时，自动检查 `review-schedule.md` 中是否有今日到期项
- 今日到期项按**拓扑层级 + 到期紧急度**双因素排序（层级低的先复习——那是后续知识的基础）
- 今日到期项 ≤ 3 道：逐道完成复述 + 评分
- 今日到期项 > 3 道：优先复习拓扑层级最低 + 评分最低的 3 道，其余顺延到次日
- 每日复习上限默认 5 道（可配置），超过则自动顺延
- 如果某道题被跳过，且其依赖链上有更高层级的题，系统会提示："跳过这道的话，依赖它的 X 道题复习效果会打折"

**复习时的复述流程**（与模式二的复述环节一致）：

1. 展示题目（不展示参考答案）
2. 用户口述回答
3. 展示参考答案，用户对比自评（0/1/2/3）
4. 系统自动更新间隔和下次复习日期
5. 如果评分 ≤ 1，追问卡住位置诊断

### 知识依赖图谱与拓扑复习

知识不是孤立的。一道面试题卡住了，往往不是因为这道题本身难，而是因为**前置知识没掌握**。本系统为每条知识条目维护依赖关系，实现"先修基础、再攻难题"的拓扑复习顺序。

**依赖关系建模**：

每条面试题和知识条目在创建时，AI 会分析其依赖的前置知识，记录在 `knowledge-index.md` 的依赖矩阵中：

```markdown
## 知识依赖矩阵

| 条目 | 依赖（前置知识） | 被依赖（后续知识） |
|------|-----------------|-------------------|
| Q-003 PostgreSQL MVCC 原理 | KB-001 数据库事务 ACID | Q-007 PostgreSQL vs MongoDB, Q-012 慢查询排查 |
| Q-007 PostgreSQL vs MongoDB | Q-003 PostgreSQL MVCC, KB-002 MongoDB 文档模型 | Q-015 架构选型方法论 |
| KB-001 数据库事务 ACID | 无（基础知识） | Q-003, Q-007, Q-010 分布式事务 |
```

**依赖发现机制**：

- **显式依赖**：用户在复述时明确指出"这道题卡住是因为前置知识 X 不懂" → AI 记录依赖关系
- **推断依赖**：AI 分析题目内容，自动推断可能的依赖关系（如"Redis Cluster 方案对比题"依赖于"Redis 单机数据结构的理解"），标注 `[AI推断]` 并在下次复习时向用户确认
- **卡住溯源**：当用户在多道题上卡住且诊断均为同一类型时，AI 追溯是否为同一前置知识缺失

**拓扑复习排序**：

复习调度不再只是按"到期日期"排列，而是按**依赖拓扑 + 到期日期**双因素排序：

```
复习优先级 = 拓扑层级（先复习被依赖的） + 到期紧急度（快到期的优先）

示例：
  KB-001 数据库事务 ACID — 层级 0（无依赖）— 🔴 今日到期 → 排第 1
  Q-003 PostgreSQL MVCC — 层级 1（依赖 KB-001）— 🟡 明日到期 → 排第 2
  Q-007 PostgreSQL vs MongoDB — 层级 2（依赖 Q-003）— 🟢 3天后到期 → 排第 3
```

**复习时的依赖感知提示**：

每次展示题目时，如果该题有未掌握的前置知识，首先显示警告：

> "⚠️ 这道题依赖 2 个前置知识点，其中 1 个你还没掌握：
> - Q-015 PostgreSQL MVCC 原理 — 🔴 未复述
> 
> 建议先复习 Q-015，再回来做这道题。或者你觉得自己能直接讲？"

**依赖链自动检测**：

每次新增题目时，AI 自动检查：
1. 新题的前置知识是否已存在于知识库中 → 建立链接
2. 新题是否可以作为已有题目的前置知识 → 反向建立链接
3. 是否有循环依赖（A 依赖 B，B 依赖 A）→ 自动拆解为并列关系

**依赖已掌握后的连锁更新**：

当用户将某道题评分提升到 3（流畅复述）时，AI 自动检查所有依赖它的后续题目，更新它们的"前置知识链"状态，并在下次复习时提示：

> "Q-015 PostgreSQL MVCC 已掌握！这意味着依赖它的 Q-007（PostgreSQL vs MongoDB）和 Q-012（慢查询排查）的前置障碍已经清除——要不要趁热打铁，把这两道也复习了？"



知识条目有生命周期。随着技术栈升级和经验增长，部分条目会过时。必须定期清理，避免知识库变成"过期信息的垃圾场"。

**条目元数据**（每条知识条目必须记录）：

```markdown
- **创建日期**：2026-06-24
- **技术栈版本**：Python 3.11 / React 18 / Docker 24
- **归档状态**：活跃 / 已过时 / 已替代 / 仍有效
- **最后审查日期**：未审查 / 2026-09-24
```

**审查触发条件**：

| 条件 | 行为 |
|------|------|
| 条目创建超过 90 天 | 提醒用户审查是否仍有效 |
| 条目创建超过 180 天 | 加强提醒，建议强制审查 |
| 条目创建超过 365 天 | 自动标记为 `[待年审]`，需用户在下次会话中逐条确认 |
| 用户主动请求 | 立即启动指定项目/时间段条目的审查 |

**归档标记**：

| 标记 | 含义 | 后续处理 |
|------|------|----------|
| `[活跃]` | 仍在学习/复习中 | 正常参与复习调度 |
| `[已过时]` | 技术已过时，不再适用 | 移入 `archive/`，不参与复习 |
| `[已替代]` | 被更好的方案取代 | 移入 `archive/`，保留作为对比参考 |
| `[仍有效]` | 超过 90 天但依然正确 | 更新审查日期，继续活跃 |

**归档文件结构**：

```
knowledge/archive/
├── archived-qa.md           # 已过时的面试题
├── archived-pitfalls.md     # 已过时的避坑条目（技术已不存在等）
└── archive-log.md           # 归档记录（时间、原因、操作人）
```

**季度审查流程**：

每季度首次使用 Skill 时触发：

1. 列出所有超过 90 天未审查的活跃条目
2. 逐条询问：`[仍有效]` / `[已过时]` / `[已替代]` / `[稍后审查]`
3. 将标记为过时/已替代的条目移入 `archive/`
4. 更新 `knowledge-index.md` 中的统计数字

### 模式四：快问快答 — 单一决策点快速深挖

**目标**：针对一个技术决策，5 分钟内完成"追问到底 → 生成 1 道深挖原理题 → 骨架复述"。不需要完整的对话记录，不需要分阶段复盘。

**适用场景**：
- 用户刚跟 Agent 做了一个技术决策，趁热打铁想弄懂为什么
- 用户在复习时发现某道题卡住，想针对那个知识点快速补上
- 用户在面试前想突击搞清几个关键概念
- 用户只有 5 分钟，不想启动完整的复盘流程

**触发方式**：
- "快问快答：[技术决策]"（如"快问快答：为什么用 Redis 而不是直接查数据库"）
- "帮我深挖一下 [知识点]"
- "5 分钟搞懂 [概念]"

**处理流程（三步闭环）**：

**第一步：追问到底**

用户提出一个技术决策或知识点后，AI 不直接给结论，而是追问三层：

```
层级 1 — 做了什么：Agent 选了 X 而不是 Y。（用户提供）
层级 2 — 为什么：X 和 Y 的本质区别是什么？什么场景下选 X？什么场景下选 Y？
层级 3 — 凭什么：支撑这个选择的底层原理是什么？如果条件变了，选择还成立吗？
```

AI 会追问用户以确认推理方向：

> "Agent 选了 Redis 做缓存。在你这个场景中，Redis 替代的是什么？是数据库直读对吗？那如果不考虑性能，数据库直读和 Redis 缓存的本质区别是什么？"

**第二步：生成 1 道深挖原理题**

基于三步追问的结果，生成 1 道面试题（格式与模式二的深挖原理题一致，包含场景描述、问题、答案要点、反事实推理、深挖追问）。

**第三步：骨架复述**

遵循分层复述规则——要求用户 30 秒口述核心思路（必做），然后邀请完整复述（选做）。评分和即刻补刀机制与模式二一致。

**知识库沉淀**：

快问快答生成的题目同样追加到知识库（`interview-qa.md`），参与间隔复习调度。题目来源标注为 `[快问快答]`。

**与完整复盘的区别**：

| 维度 | 完整复盘（模式一~三） | 快问快答（模式四） |
|------|---------------------|-------------------|
| 输入 | 完整对话记录 | 单一技术决策/知识点 |
| 耗时 | 15-30 分钟 | ≤ 5 分钟 |
| 产出 | 5-15 道题 + 避坑清单 + 知识条目 | 1 道深挖原理题 |
| 提取维度 | 5 维度全面扫描 | 聚焦单一维度 |
| 适用时机 | 项目阶段性复盘 | 随时触发、碎片化学习 |
| 门槛 | 需要有完整对话 | 只需一个具体问题 |

## 防御机制汇总

本 Skill 内置以下防御机制，确保输出质量：

| 机制 | 触发条件 | 行为 |
|------|----------|------|
| **用户确认** | AI 推断的决策理由 | 标注 `[待确认]`，反问用户确认 |
| **范围限制** | 对话超过 50 轮 | 建议分段复盘，每次只处理一个阶段 |
| **分层复述** | 每道题生成后 | 骨架复述（必做，30s 核心思路）+ 完整复述（选做，2min 细节），降低开口门槛 |
| **即刻补刀** | 复述卡住时 | 根据诊断类型（概念不清/逻辑链断裂/细节遗忘）当场生成 2 分钟微学习内容，补刀后可重新尝试复述并提分 |
| **渐进式提醒** | 连续跳过骨架 ≥3 次 | 主动降速提醒；连续只做骨架 ≥3 次温和引导完整复述；完全跳过 ≥10 次硬性暂停 |
| **种子对话** | 新用户无对话记录 | 提供预置的 18 轮示例对话，可立即体验完整三模式流程，零门槛上手 |
| **快问快答** | 单一决策点、碎片化时间 | 5 分钟三步闭环（追问到底 → 生成 1 道深挖原理题 → 骨架复述），无需完整对话 |
| **信号过滤** | 提取知识点时 | 丢弃重复操作、无意义对话、纯机械操作 |
| **深度标注** | 知识点深度不足时 | 标注 `[待深挖]`，提醒用户进一步学习 |
| **去重合并** | 增量更新知识库时 | 检测相似条目，合并而非重复添加 |
| **过期审查** | 知识条目超过 90 天 | 提醒用户审查有效性，标记为过时/替代/仍有效，过时条目移入 archive/ |
| **依赖拓扑** | 题目创建时 / 复述卡住时 | 分析并维护知识依赖关系；复习按拓扑层级排序（先复习被依赖的基础知识）；卡住时溯源前置知识缺失；前置掌握后连锁提示后续题目 |
| **冷启动引导** | 用户首次使用且无对话记录 | 提供最小启动步骤（含种子对话入口），不硬凑知识点 |

## 边界情况处理

**输入过短**：如果用户只提供了简短描述（少于 5 轮对话），主动追问以补全上下文。至少需要了解：项目是什么、做了什么、遇到了什么问题。

**输入过长**：优先提取最有价值的知识点。如果覆盖多个技术领域，按领域拆分为独立的输出文件。主动询问用户最关注哪些维度。

**无明显学习点**：如果 Agent 只是做了简单直接的工作（如 CRUD 模板生成），诚实告知用户"本次会话学习点有限"，建议用户选择更复杂的会话进行复盘。不要硬凑知识点。

**多技术栈混合**：按技术栈拆分为独立的知识条目，每个条目聚焦单一技术点。在索引中标注跨技术栈的关联。

**用户是初学者**：如果提取的知识点超出初学者理解范围，在输出中增加"前置知识"说明，帮助用户建立学习路径。
